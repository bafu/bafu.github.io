## TL;DR

自社構築を決めた後、4日間でゼロから動くプロトタイプまで駆け抜けました。Lovableがフロントエンド、Claude Codeがバックエンド、私がアーキテクチャの判断を担当しました。最も重要だった定義は3つ——Claude Codeを設計図としつつWeb型のアーキテクチャにすること、per-user（ユーザー単位）ではなくper-conversation（会話単位）のワークスペース分離でチーム協業を可能にすること、そして「まず動かす」と「使えるようにする」を明確に分けること。これらの判断はホワイトボードの前で考え出したものではありません。AIが期待通りに動かない、UIが崩れる、接続が切れる——その混乱の中で、一つずつぶつかりながら見つけたものです。

---

## 出発点：AIのために書いた仕様書

W1では、なぜ自社で構築するのかについて語りました。でも「なぜ」と「どうやって」の間には、大きな溝があります。

1月3日の午前9時半、振り返ってみて初めて重要だったと分かることをしました。実装計画（implementation plan）を書いたのです（`da2c16c`）。

人間のために書いたものではありません。AIのために書きました。

| Claude Code コンポーネント | Numbers Omni 対応 |
|---|---|
| CLIインターフェース | Web UI（Lovableフロントエンド） |
| アプリケーションオーケストレーター | FastAPI Gateway + WebSocket |
| Agentic Loop | Claude Agent SDK（Python） |
| Tool Runtime | ローカル実行環境（GCE VM） |
| Project Context | Workspace Manager |

この文書は、Claude Codeのすべてのコンポーネントを、Omniの対応物に一対一でマッピングしたものです。システムアーキテクチャ、ファイル構造、実装フェーズ、ツール定義、WebSocketメッセージプロトコル、セキュリティ境界——すべてが含まれています。

> **エンジニア脳：** 世界最高のAI開発ツールはClaude Codeです。
> AI主権の問題がなければ、私はClaude Codeを使い続けていたでしょう。
> 自社で構築する以上、巨人の肩の上に立つべきです——
> AIにClaude Codeを徹底的に研究させ、そのアーキテクチャを再現する。

> **ファウンダー脳：** でも、私が欲しいのは「もう一つのClaude Code」ではありません。
> 既存のチームとシームレスに統合でき、技術職でない同僚も使え、
> 8人分のサブスクリプションより安く済むもの。
> Claude Codeは個人ツールです。Omniはチームツール。根本的に違います。

この点では、二つの脳が一致しました。Claude Codeのアーキテクチャを出発点にしつつ、すべての設計判断を「チーム協業」の方向に修正していく。

家を建てるのと同じです。レンガを再発明する必要はありませんが、設計図は自分で描く必要があります。Claude Codeはレンガの仕様書。Omniの設計図は私自身が描いたものです。

## 2つのAI、1人の人間

実装計画を書き終えてから23分後（09:55）、FastAPIのバックエンド全体が稼働しました（`3fb0958`）。

私が書いたのではありません。Claude Codeが書きました。WebSocketゲートウェイ、自作のReActエージェントループ、5つのコアツール（bash、read_file、write_file、edit_file、glob）、Supabase JWT認証、ワークスペース分離——すべてが1つのコミットに入っています。

ただし、バックエンドはストーリーの半分に過ぎません。

同じ日の午前7時、Lovableはすでにフロントエンドの基盤を完成させていました。Supabase接続、認証システム、データベーススキーマ、3カラムUIフレームワーク。10時半にはフロントエンドとバックエンドの接続が完了。11時にはCloudflare TunnelでWSS接続が開通しました。

**初日で動くプロトタイプが完成しました。**

> **エンジニア脳：** この速度は従来の開発では不可能です。
> 理由は、フロントエンドとバックエンドを2つの異なるAIツールに任せたから——
> LovableがReact + Supabaseのフロントエンドスタック全体を担当し、
> Claude CodeがPython + FastAPIのバックエンドを担当。
> 2つのラインが並行して進み、私は真ん中で判断を下しました。

> **ファウンダー脳：** 技術的にはすごい。でも「動く」と「使える」の距離は、
> 想像していたよりずっと遠かった。

## クラッシュ日記

1月3日から1月6日。4日間で138コミット。Lovableの自動同期を除くと、実質的な機能追加やバグ修正は約90件。1日あたり20件以上。

これは通常の開発ペースではありません。あの4日間の実態をお伝えします。

**1月3日——すべてが順調に見えた。そして順調でなくなった**

09:55にバックエンドが稼働、11:20にCloudflare Tunnelが接続。そして11:48、最初の壁にぶつかりました。JWT認証が失敗——Supabaseが使っていたのはES256アルゴリズムで、私たちが想定していたRS256ではありませんでした。同時に、CORSがWebSocketをブロック。夜9時になっても、まだWebSocketのデバッグログを追加していました。

**1月4日——接続はすべての基盤。なのに切れ続ける**

07:10にWebSocketのハートビート機構を追加しました（15秒間隔）。Cloudflare Tunnelは、長時間データがないと接続を切断するためです。11分後（07:21）、より深い問題が見つかりました。同期版のAnthropicクライアントが、レートリミットのリトライ時に`time.sleep()`を使い、イベントループ全体をブロックしていたのです。ハートビートが送信できず、Cloudflareが接続を切断しました。解決策は`AsyncAnthropic`への切り替えでした。

1つのバグが2つ目のバグを引き出し、2つ目のバグがアーキテクチャの問題を露呈させる。あの4日間は毎日この繰り返しでした。

その後はモバイル対応、GCPクレデンシャルの統合、GitHubのREST APIからCLI + MCPへの切り替え。どれも「やってみて初めて、うまくいかないと分かる」ものでした。

**1月5日——アーキテクチャレベルの振り返り**

07:24に最も重要なアーキテクチャ変更を行いました。ワークスペースをper-user（ユーザー単位）からper-conversation（会話単位）へ変更（`f5a61d8`）。午後にはマルチファイルタブ（`ec49a71`）、AWSとFirebaseのクレデンシャル対応、プロンプトキャッシングを有効化しました。

**1月6日——ツールの大爆発**

一気に6つの新ツールを追加。TodoWrite、Grep、AskUserQuestion、WebSearch、WebFetch、Task（サブエージェント生成）。同時に、クレデンシャル漏洩のバグも修正しました——エージェントのレスポンスとキャンバスログがAPIキーを露出させていたのです。PWA対応、チャット共有機能、バージョン表示も追加しました。

> **ファウンダー脳：** あの4日間、計画はゼロでした。
> 醜いUI、ずれたボタン、すぐに切れる接続——見るたびに心が折れそうでした。
> 「もう続けるべきか？やめたほうがいいのか？」という声が常にありました。
> でも諦めきれない。「もう一回だけ、本当に最後の一回」。
> そうやって、眠らずに4日間を過ごしました。

> **エンジニア脳：** 2つのAIに同時に引っ張られている感覚でした。
> Lovableは数分おきに新しいUI変更を出し、Claude Codeは数分おきに新しいバックエンドコードを出す。
> AIよりも速く判断を下さなければ、間違った方向に突き進んでしまいます。
> 止まることはできません。止まるコストのほうが、壁にぶつかるコストより高いからです。

2台のランニングマシンに同時に乗っている状態を想像してください。どちらも加速しています。降りることはできません。降りたら転ぶからです。できるのはリズムを合わせて、ついていくことだけです。

## 最初のアーキテクチャ判断：per-conversation

1月5日。プロジェクト開始からまだ2日目。最初のアーキテクチャレベルの変更を行いました。

元の設計は直感的でした。ユーザーごとに1つの作業ディレクトリ（`/workspaces/{user_id}/`）。一人ひとりに自分の机がある、というイメージです。

2日間の実使用で、これではうまくいかないと気づきました。

```python
# 旧：ユーザーごとに1ディレクトリ
/workspaces/{user_id}/

# 新：会話ごとに1ディレクトリ
/workspaces/conversations/{conversation_id}/
```

> **エンジニア脳：** per-userの問題は明確でした——
> 同じユーザーの異なる会話が互いに汚染し合います。
> 会話Aでファイルを変更すると、会話Bのエージェントが文脈のない変更を見ることになります。
> より根本的な問題は、異なるユーザーが同じ会話で協業できないことです。

> **ファウンダー脳：** 企業AI技術コンサルタントとして働いていた時に気づいたことがあります。
> 技術職でない社員の多くは、実はvibe codeをやりたいと思っています。でも、期待と不安が入り混じっている。
> 法務は「自分がvibe codeで作ったツールにセキュリティの脆弱性があったら」と心配します。
> 技術的な責任を取れないという恐怖が、自分の業務に必要なツールを作ることを阻んでいます。
> 開発者が任意の会話に簡単に入って、非技術チームの環境を調整できれば、
> 全員がもっと安心してAIを使えるようになります。

per-conversationはパスの1行の差に過ぎません。しかし、それ以降のOmniのすべての設計方針を定義しました。**すべてを会話単位に、すべてをチーム協業のために。**

セッションモデルも変わりました。WebSocket接続はワーキングディレクトリにバインドされなくなりました。ユーザーが会話を切り替えると、バックエンドが`switch_workspace`メッセージでワークスペースを動的に切り替えます。切断も再接続も不要です。すべてのツール操作は、接続時に固定するのではなく、セッションから現在のワークスペースパスを動的に読み取ります。

オフィスのデザインに例えると——一人ひとりに個室を与えるのではなく、プロジェクトごとに会議室を用意するようなものです。誰でも入って、すべての資料を見て、作業を引き継ぐことができます。

## システム定義：何をやるか、何をやらないか

4日間が終わった後、システムの輪郭が明確になりました。事前に計画していたからではありません。壁にぶつかるたびに、修正するたびに、アーキテクチャを変更するたびに、同じ問いに答えていたからです。**私たちには、実際にどんなシステムが必要なのか。**

答えは2つのパートに分かれます。

**Phase 1——まずシステムを動かす：**

| 判断 | 選択 |
|---|---|
| バックエンドフレームワーク | FastAPI + WebSocket（リアルタイムストリーミング、request-responseではない） |
| AIエンジン | 自作のReActエージェントループ + 5つのコアツール |
| 認証 | Supabase JWKS公開鍵検証（ES256） |
| 接続 | Cloudflare Tunnel + ハートビート機構（IPを公開しない） |
| 分離 | Per-conversationワークスペース |
| フロントエンド | Lovable（React + TypeScript + Supabase） |

**Phase 2——使えるようにする（意図的に後回し）：**
- MCPクライアント（外部ツール統合）
- Git統合（自動コミット、差分分析）
- プレビューサーバー（ユーザーアプリケーションのライブプレビュー）
- コンテキスト要約（トークン管理、自動要約）

> **エンジニア脳：** これはエンジニアにとって最も難しい判断です——やるべきことを知りながら、やらないことを選ぶ。
> MCP、Git、Previewは、Claude Codeにはある機能です。実装しないと不完全に感じます。
> でもPhase 1の目標は完全であることではなく、生存することです。
> 接続でき、エージェントが動き、ワークスペースが分離されたシステムのほうが、
> すべてがあるのに何も安定しないシステムよりはるかに価値があります。

> **ファウンダー脳：** それに、もう4日間寝ていません。
> ここで止めなければ、システムが定義される前に、自分のほうが先に定義されてしまいます。

この4日間で、後にプロダクト要件書に書き込まれる3つの基本原則も芽生えました——当時はまだPRDが存在していませんでしたが。

1. **Claude Code Web Parity**：Claude Codeの能力を基準としつつ、Webで実装する——開発中の接続断を許容できないため、ターミナルではなくWebである必要があります
2. **Internal Knowledge First**：社内リソース（リポジトリ、クラウド設定、ログ）を優先的に活用する。外部検索は最後の手段です
3. **Incremental Capability**：すべての機能は増分的に追加する。既存の機能を削除しない——Phase 2のすべての項目は積み上げであり、作り直しではありません

これらの原則はホワイトボードの前で考え出したものではありません。あの4日間の混乱の中で、何度も選択を繰り返した末に浮かび上がったパターンです。

## 今週の結論

システムを定義するには、3つのものが必要です。

1. **良い参照物。** Claude Codeは当時最高のAI開発ツールでした。ゼロから設計するよりも、その肩の上に立ち、「どこを変えるか」にエネルギーを集中させるべきです。CLIではなくWeb、個人ではなくチーム、per-userではなくper-conversation。

2. **譲れない核心的な違い。** Omniにとって、それは「チーム協業」です。Per-conversationワークスペースは技術的な好みではなく、プロダクトとしての信念です。技術職でない同僚が安心してvibe codeでき、技術チームがいつでも介入して監督できること。

3. **壊れても続ける人間。** AIはコードを書けます。でも、午前3時に「このアーキテクチャは間違っている。変更しなければ」と決断することはできません。あの4日間で最も重要な成果は、138のコミットではありません。クラッシュの中で検証された、一連の判断です。

4日間でシステムを定義できるか？振り返ってみると、答えはこうです。4日間でシステムの**境界**を定義できます。何であるべきか、何であるべきでないか、そしてどの課題を未来の自分に残すべきか。それが分かります。

家を建てる最初の4日間と同じです。壁を積んでいるつもりが、実は基礎の位置を確認している。壁は壊して積み直せます。でも基礎が決まれば、建物全体がそれに従います。

## 対談：Agentとの問答

**Agent：4日間で138コミット。Lovableがフロントエンド、Claude Codeがバックエンド、2つのAIを同時に操作。振り返ってみて、あの4日間はどんな状態でしたか？計画的なスプリントでしたか、それとも止められない状態でしたか？**

振り返ってみると、「AIに駆動されている」モードに入っていたように思います。基盤を構築していた初期の段階で、UIからバックエンドまで頭の中にはたくさんのアイデアがありました。でもAIが思い通りに作ってくれない。醜いUI、ずれたボタン、すぐに切れる接続を見るたびに、本当に心が折れそうでした。「まだ続けるべきか？もう諦めたほうがいいのか？」という声が常にありました。でも諦めきれない。「もう一回だけ、本当に最後の一回」と自分に言い聞かせ続けて、気づいたら眠らずに4日間が過ぎていました。計画はゼロでした。振り返ってみれば、まったく新しい開発体験だったと思います。

---

**Agent：実装計画ではClaude Codeの各コンポーネントをOmniに一対一でマッピングしています。なぜ「Claude Codeのアーキテクチャを複製する」ことを出発点に選んだのですか？企業ニーズから新しいシステムを設計するのではなく。**

Omniを設計していた当時、世界で最も優れた開発ツールはClaude Codeでした。会社のAI主権やコントロール、チーム協業を重視しなければ、私は間違いなくClaude Codeを使い続けていたでしょう。Claude Codeがそれほど優れているなら、同じくClaude SDKをベースにしたOmniも巨人の肩の上に立つべきです。そこで、AIにClaude Codeを徹底的に研究・学習させることにしました。この記事を書いている時期にちょうどClaude Codeのソースコードが流出しましたが——Omniを開発していた時にソースコードがあったら、たぶん使いたかったですね。残念です（笑）。

---

**Agent：プロジェクト3日目で、ワークスペースをper-userからper-conversationに変更しました。なぜこんなに早い段階で、per-userの分離では不十分だと気づいたのですか？**

協業と安全性です。OmniとClaude Codeの最も核心的な違いは、チームツール vs. 個人ツールだと考えています。企業AI技術コンサルタントとして活動していた中で気づいたことがあります。技術職でない社員の多くは、実はvibe codeをやりたいと思っています。でも、期待と不安が入り混じっている。例えば、法務は「自分がvibe codeで作った法務ツールにセキュリティの脆弱性があったら」と心配します。技術を理解していない、技術的な責任を取れない——その恐怖が、自分の業務に必要なツールを作ることを阻んでいます。これは個人にとっても会社にとっても損失です。だからOmniを設計する時に考えたのは、既存の開発者が法務や人事など非技術チームのAI開発環境を簡単に調整でき、問題があれば技術チームがすぐに引き継げる仕組みを作ること。技術を理解している人がAIを監督する協業環境を作れば、全員が安心してAIでvibe codeを使い、個人と会社の生産性を大幅に向上させられます。この目標のために、per-conversationはper-userより自然な選択でした。
