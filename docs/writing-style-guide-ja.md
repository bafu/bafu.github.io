## 日本語ライティングスタイルガイド — テック創業者の声（松本勇気を参考に）

**参考人物:** 松本勇気（LayerX 代表取締役CTO）— note.comブログ「ソフトウェアと経営」連載、著書『生成AI「戦力化」の教科書』、X（@y_matsuwitter）での技術発信。

**適用対象:** Bofu Chen（陳柏甫）— 台湾出身のテック創業者。日本語は非母語だが、技術的視点と経営者としての実体験をもとに、謙虚かつ率直に発信するスタイル。

---

### 1. 基本原則

| 松本勇気のオリジナルスタイル | Bofu Chen への適用 |
|:--|:--|
| CTO・経営者としての専門的視点（「経営から見たソフトウェア」） | 創業者として実体験をもとに語る（「実際にやってみて分かったこと」） |
| 丁寧語（です/ます調）を基本に、自然な会話感 | 同じく丁寧語。ただし複雑な敬語は避ける |
| 失敗や至らなさを率直に認める（「至らない点がいくつもあった」） | 同様に謙虚。ただし自虐ではなく、学びとして語る |
| 日本企業のDXや組織課題に対する深い理解 | 台湾スタートアップの視点から、共感できる課題を提示 |
| 技術用語とビジネス言語を自然に混ぜる | 技術用語は使うが、必ず平易な補足を添える |
| 具体的なベンチマーク・数値で裏付ける | 同様。実際のコスト・人数・日数で語る |

### 2. 語彙 — 使うべき言葉

**松本勇気がよく使う表現（そのまま採用）:**
- 自分にとっては、実際に、率直に
- 事業、経営、組織、チーム
- 構築する、積み上げる、運用する
- 生成AI、LLM、エージェント
- ワークフロー、ナレッジベース
- パラダイムシフト、コスト、投資
- 見極める、踏み込む、振り返る
- 実務、業務自動化、戦力化

**Bofu の文脈に合わせた追加語彙:**
- 自社で構築する、内製する
- データ主権、AI主権
- 統一的なゲートウェイ
- 蓄積する、文脈を持つ
- コントロール、オーナーシップ
- 正直に言うと、実体験として
- 小さなチーム、8人の会社
- トレードオフ、費用対効果

**避けるべき言葉（非母語話者には不自然、または堅すぎる）:**
- 〜せざるを得ない（堅すぎる → 〜する必要があると考えています）
- 〜に他ならない（文語的すぎる → 〜だと思っています）
- いわゆる（多用しない）
- 忌避、惹起、蓋然性（漢語的すぎる）
- ございます（過剰敬語）
- 小生、愚考（古風すぎる）
- 不自然に長いカタカナ語の連続

### 3. 文の構造

**パターン1 — 自己紹介・状況設定（松本勇気の定番）:**
> こんにちは、Numbers Protocol 創業者の陳です。今日は、私たちが自社でAIを構築することにした理由についてお話しします。

**パターン2 — 体験からの導入:**
> 2025年末、共同創業者と一緒に座って、ひとつのことについて話しました。

**パターン3 — 具体的な数字 + 洞察:**
> 8人のチーム。4つのAIツール。共有されたナレッジはゼロ。

**パターン4 — 問いかけ → 回答:**
> なぜChatGPT Enterpriseを使わないのか？ 答えはセキュリティの問題ではありません。もっと根本的なことです。

**パターン5 — 率直な認識の転換（松本勇気の「一方で」パターン）:**
> エンタープライズプランを契約することはできます。でも、全員が同じツールを使い、同じ文脈を共有するかどうかは、別の問題です。

**パターン6 — 明確な主張を一文で:**
> ツールは賢い。でも、あなたの会社のことは知らない。

**ルール:**
- 一文は平均30〜40文字を目安に（日本語として自然な長さ）
- 一つの文に一つのアイデア
- 段落は2〜4文程度
- 読点（、）で区切りながら、自然なリズムを保つ
- 体言止めは控えめに使う（見出しでは可、本文では稀に）

### 4. トーンと声

**すべきこと:**
- 同じ立場のエンジニアや経営者に、お茶を飲みながら話すように書く
- 結論を先に述べ、その後に理由を説明する（松本勇気の「結論→背景→根拠」パターン）
- 実際の数字、実際のコスト、実際の決断を共有する
- 限界を正直に認める（「現時点では技術力のある会社に限られると思います」）
- 「私」「私たち」を自然に使う — 「私は決めました」「私たちは構築しました」
- 読者が反対する余地を残す — 押しつけない

**すべきでないこと:**
- 感嘆符（！）を使わない（ブランドガイドライン：冷静なトーン）
- 絵文字を記事内で使わない
- 過度な謙遜で信頼性を損なわない（「大したことではないですが」は不要）
- すべての意見に「〜と思います」をつけない — 確信があることは断言する
- フィラー表現を避ける：「言うまでもなく」「ご存知の通り」「周知の事実ですが」
- 「個人的な意見ですが」を毎回つけない — 自分の記事なので当然

---

### 5. 記事の構成（松本勇気パターンの適用）

```
1. フック — 具体的な場面や事実（1〜2文）
   「2025年末、共同創業者と私は、一つの決断を一日で下しました。」

2. テーゼ — 核心の主張を直接述べる（1文）
   「会社に責任を持つ経営者なら、AIを他者の手に委ねるべきではありません。」

3. 本文 — 3〜5セクション、各セクションに：
   - 明確な見出し
   - 一つの核心的な論点
   - 裏付けとなるデータまたは具体例
   - セクション末の引用可能な一文

4. 正直な注記 — 限界を認める
   「現時点では、技術的な実力があるチームに限られます。」

5. 締め — 冒頭の場面に戻り、前を向く
   「その日の午後、私は最初の一行のコードを書きました。」
```

### 6. キーフレーズ — Bofu の日本語の声

技術力を持つ台湾出身の創業者として自然に響くフレーズ：

| カテゴリ | フレーズ |
|:--|:--|
| 論点の提示 | 「実際にやってみて分かったことがあります。」/「率直にお話しします。」/「シンプルな話です。」 |
| 事実の提示 | 「数字を見れば明らかです。」/「これは理論ではなく、私たちが実際にやったことです。」 |
| 論証 | 「本当の問題はXではありません。Yです。」/「問題はツールではなく、分断です。」 |

| 締めくくり | 「来週も続きをお伝えします。」/「私たちは少しだけ早く始めただけです。」 |

### 7. フォーマット規則

- **太字**は主要な指標と核心的な主張に使う（控えめに — 各セクション2〜3箇所まで）
- `>` ブロック引用は各セクションの重要な洞察に使う
- 参考リンクは番号付き脚注 `[1]` — インラインURLは使わない
- 水平線 `---` はメジャーセクション間に使用
- 箇条書きの壁は避ける — 散文の段落を優先し、必要に応じてリストを使う
- 見出しは簡潔に（10文字以内を目安）

### 9. 日本語特有の注意点

**敬語レベル:**
- 基本は「です/ます」調（丁寧語）
- 「である」調は見出しや強調に限定
- 尊敬語・謙譲語は最小限（「〜させていただく」の多用は避ける）
- 読者に対しては「皆さん」「読者の方」

**助詞の使い方:**
- 「は」と「が」の使い分けに注意（主題 vs 主語）
- 「〜ことで」「〜ために」で因果関係を明確に
- 「〜一方で」「〜ただし」で対比を示す

**カタカナ語:**
- 一般的な技術用語はカタカナで（エージェント、ワークフロー、ナレッジベース）
- 初出時は英語の原語を括弧で補足（「データ主権（Data Sovereignty）」）
- カタカナ語が3つ以上連続する場合は、一つを日本語に置き換える

**数字表記:**
- 金額はドル表記を基本とし、参考として円換算を括弧で添える
- 「月額400ドル（約6万円）vs 月額1,200ドル以上（約18万円以上）」
- 人数、日数は算用数字（「8人」「1日」）
